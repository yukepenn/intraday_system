1. High-level architecture
The new system should be built around these layers:
Layer 0 = data, calendar, feature store, signal store, cache system
Layer 1 = candidate factory
Layer 2 = candidate competition, router, management, portfolio behavior
Layer 3 = frozen system validation
Reference engine = readable truth path
Fast engine = NumPy/Numba acceleration path
The most important rule:
Reference Python is truth.
Numba is acceleration.
Never allow two different PnL truths.
This means we design both engines from the beginning, but the fast engine must always be parity-tested against the reference engine.
A simple mental model:
Parquet bars
→ curated session-aware bars
→ NumPy BarMatrix
→ cached FeatureMatrix
→ cached SignalMatrix
→ TradeIntent
→ execution reference/fast
→ TradeRecord
→ metrics
→ candidates
→ router/combiner

2. Why this architecture is right
We want the repo to make future research easy:
Add new feature    → write feature kernel + config + test
Add new strategy   → write signal function + base config + grid + test
Add new sweep      → write YAML config, not new engine code
Add new router     → write router rule/config, not new PnL logic
Add new management → write management builder, not new backtest engine
The system should not require custom scripts everywhere.
Most future work should be done through:
intraday layer1 run --config ...
intraday layer1 promote --run-id ...
intraday layer2 run --config ...
intraday layer3 run --config ...
intraday validate parity
That is the centralized style we want.

3. External framework ideas to mimic
We should not fully adopt vectorbt, LEAN, backtrader, or Zipline, because your project has a specialized 1-minute intraday research architecture. But we should borrow specific ideas.
From vectorbt, mimic the array-first model: signals, orders, trades, logs, and metrics should be represented as structured arrays, with Numba used for speed-sensitive simulation. Vectorbt’s portfolio package is built around portfolio modules and Numba-backed internals, which is the right performance mindset for large sweeps. (VectorBT)
From QuantConnect LEAN, mimic modularity: data, algorithm logic, execution, portfolio, brokerage, and result handling are separate engine components. LEAN is described as an open-source algorithmic trading engine for research, backtesting, and live trading. (QuantConnect)
From backtrader, mimic the “central orchestrator” idea. Backtrader’s Cerebro gathers data feeds, strategies, observers, analyzers, writers, executes backtesting/live flow, and returns results. We should create our own cleaner intraday equivalent through a central CLI/runner, not through scattered scripts. (backtrader.com)
From Zipline, mimic event discipline and calendar-aware backtesting. Zipline is described as a Pythonic event-driven backtesting system, and event discipline is useful for intraday session boundaries, EOD handling, and next-bar entry semantics. (GitHub)
For data access, keep Parquet as cold storage and use Polars or DuckDB for efficient scans. Polars scan_parquet supports lazy reads with predicate/projection pushdown, and DuckDB’s Parquet reader supports pushing filters and projections into scans. (Polars User Guide)
For hot computation, use NumPy arrays + Numba. Numba’s performance guidance explicitly supports writing loop-style numerical code when compiled properly, which matches trade-path simulation very well. (Numba Documentation)

8. Cache system
The cache system is central. It should prevent repeated computation.
8.1 Cache layers
data cache      = BarMatrix arrays
feature cache   = FeatureMatrix
signal cache    = SignalMatrix per candidate
Layer2 cache    = candidate signal matrix + router-ready arrays
metrics cache   = optional summary outputs
8.2 Cache keys
Every cache key should be deterministic:
data_hash
feature_config_hash
strategy_config_hash
execution_spec_hash
candidate_hash
router_config_hash
layer2_config_hash
fold_hash
Example:
cache_key = hash_config({
    "symbol": "QQQ",
    "start": "2023-01-01",
    "end": "2024-12-31",
    "data_hash": bars.data_hash,
    "feature_config": feature_config,
})

31. Summary of the new system
The new project should be:
centralized
config-driven
array-first
cache-first
reference-truth
Numba-accelerated
Layer1/Layer2/Layer3 clean
strategy logic separated from execution
execution separated from router
router separated from validation
YAML runtime truth
CSV/MD audit only
The architecture in one line:
data → features → strategies → execution → Layer1 candidates → Layer2 router/management → Layer3 frozen validation
And the operating philosophy:
Start small.
Build vertical slices.
Keep contracts strict.
Cache everything expensive.
Use reference for truth.
Use Numba for speed.
Never let research artifacts become runtime truth.
This is the clean foundation I would build for intraday_system/.
